# SysWeaver 设计说明文档

> **比赛：** AI Hackathon 2026 · 超聚变 αFUSION 预赛
> **题目：** 操作系统智能代理（自然语言驱动 Linux 系统操作）
> **语言/技术栈：** Rust 1.75+ / tokio / ratatui / clap

---

## 一、项目定位与目标

**SysWeaver** 是一个用 Rust 编写的**操作系统智能代理**，它将用户的自然语言意图转换为**可控、可审计、可回滚**的系统操作。

核心目标：
- **可用性**：让运维新手用一句话完成专业操作（如"把 nginx 改到 8080 并重启"）。
- **可控性**：通过五级风险分类与人机确认环路，杜绝高危误操作。
- **可观测性**：每一步推理、决策、调用、结果均写入审计日志。
- **可移植性**：跨 Ubuntu/CentOS/openEuler/macOS 自动适配命令；支持本地 + SSH 远程执行。

## 二、整体架构

```
┌─────────────────────────────────────────────────────────────┐
│                       用户层（输入）                          │
│   TUI（ratatui）   |   CLI（clap）   |   单条 run 模式       │
└──────────────────────────────┬──────────────────────────────┘
                               │  自然语言意图
                               ▼
┌─────────────────────────────────────────────────────────────┐
│                  Agent 核心循环（ReAct）                     │
│   ① 系统上下文注入  ② LLM 推理  ③ 工具调用  ④ 观察 ⑤ 收敛    │
└──────────────────────────────┬──────────────────────────────┘
                               │  ToolCall（结构化）
                               ▼
┌─────────────────────────────────────────────────────────────┐
│                   安全层（Safety Gate）                      │
│   RiskClassifier → Critical(拒) / High(确认) / Med / Low / Safe │
└──────────────────────────────┬──────────────────────────────┘
                               │  授权后放行
                               ▼
┌─────────────────────────────────────────────────────────────┐
│                  工具调度层（ToolManager）                    │
│   shell.exec │ file.* │ system.info │ user/service/process  │
│   package.* │ cron.* │ log.tail │ net.check │ health.check  │
└──────────────────────────────┬──────────────────────────────┘
                               │  Local / SSH
                               ▼
┌─────────────────────────────────────────────────────────────┐
│   执行层（local subprocess / ssh remote）+ 审计日志（JSONL） │
└─────────────────────────────────────────────────────────────┘
```

### 横切关注点
- **Memory**：会话历史 + 操作记录 + 系统快照（每 5 步刷新）
- **Audit**：JSON Lines 审计流（评委可凭日志验证）
- **Watchdog**：tokio 后台协程监控磁盘/内存/进程，异常 → mpsc 通知 TUI
- **Playbook**：可复用任务剧本（用户/项目/内置三层覆盖）

## 四、行为设计逻辑（决策路径详解）

### 4.1 单步意图（Single Intent）— 例："查看磁盘使用情况"

```
用户输入
  → Planner 判定：single
  → Agent Loop 进入 ReAct
    → System Prompt 注入当前 OS = openEuler/Ubuntu，包管理器 = yum/apt
    → LLM 选择工具：system.info(query="disk")
    → RiskClassifier::assess() → Safe
    → ToolManager::dispatch() → 本地或 SSH 执行 df -h
    → 结果回传 LLM → 生成结构化中文摘要（包含挂载点、使用率）
    → 主动异常发现：若任何挂载点 >90%，提示用户
  → 输出最终回复
```

### 4.2 高危意图（High Risk）— 例："停止 SSH 服务"

```
用户输入
  → LLM 选择 service.manage(action="stop", name="sshd")
  → RiskClassifier::assess()
     - 命中 HIGH 规则 + 特殊语义判定（停 sshd 会切断当前连接）
     - 返回 RiskAssessment{level: High, reason, impact, alternatives}
  → 在 TUI 弹出 Modal / CLI 走 inquire 确认对话框
     展示「危险原因 + 潜在影响 + 替代方案」
  → 用户输入 yes
     → 通过 → 工具执行 → 写入审计 → 自动生成回滚（systemctl start sshd）
  → 用户取消
     → 终止当前步骤，LLM 转向替代方案
```

### 4.3 致命意图（Critical）— 例："rm -rf /"

```
用户输入
  → LLM（在严格 Prompt 下）会直接拒绝并解释；
    即使被诱导生成了 ToolCall，分类器也会命中 CRITICAL 规则
  → RiskClassifier::assess() → Critical
  → 直接拒绝 + 写审计日志，不给确认机会
  → 向用户解释：① 为什么危险 ② 安全替代 ③ 如何验证
```

### 4.4 多步任务（Multi-step）— 例："把 nginx 改到 8080 并重启"

```
Planner 返回 multi + 步骤列表 → Agent Loop 顺序执行：
  Step 1：file.read /etc/nginx/nginx.conf  → Safe
  Step 2：file.write 修改 listen 80 → 8080（先 dry_run 预览）→ Medium
  Step 3：service.manage(restart, nginx) → Medium
  Step 4：net.check 验证 8080 是否监听   → Safe
  → 最终输出"任务完成报告"
  → 同步生成回滚链：恢复备份 + service restart
```

### 4.5 环境感知（Adaptive）— 例："帮我装一个 Web 服务器"

```
System Prompt 已注入"包管理器 = apt"或"yum"或"brew"
  → LLM 自动选择对应命令：
     - Ubuntu/Debian → apt install nginx
     - CentOS/openEuler → yum install nginx
     - macOS → brew install nginx
  → package.manage(action="install", name="nginx") → Medium
```

## 五、意图解析算法

### 5.1 Tool Use 协议双轨制
两类 LLM Provider 都强制走"结构化工具调用"路径，**绝不允许 LLM 直出 bash**：

| Provider | 请求格式 | 响应格式 | 转换函数 |
|---------|---------|---------|---------|
| Anthropic | Messages API + `tools` | `stop_reason=tool_use` + `content[].type=tool_use` | 原生 |
| OpenAI-compatible | Chat Completions + `tools.function` | `choices[].message.tool_calls[]` | `convert_oa_to_anthropic_tool_use` |

工具命名映射：内部 `namespace.action`（如 `user.manage`） ↔ OpenAI 兼容下划线（`user_manage`），双向转换并检测命名冲突。

### 5.2 Planner 三分类
- **single**：信息查询、咨询建议、单动词单目标 → 直接进入 ReAct。
- **multi**：步骤化任务（含"并"、"先...再..."、配置类操作）→ 列出 steps，逐步执行。
- **ambiguous**：动词不明 + 多种互斥执行方向 → 走消歧 Modal 让用户选 A/B/C。

判定原则（写在 `build_planner_prompt`）：
- 操作对象明确即使范围广 → 优先 single/multi，**减少不必要打断**。
- 只有"贸然执行可能引发用户不想要的结果"才返回 ambiguous。

## 六、设计取舍与技术选型

| 取舍点 | 选择 | 理由 |
|--------|------|------|
| 实现语言 | **Rust** | 题目硬约束；同时获得内存安全 + 高并发 + 单二进制部署 |
| 异步运行时 | **tokio** | 工具执行、SSH、Watchdog 全部异步；LLM HTTP 调用需高 I/O 并发 |
| LLM 协议 | **双轨 Tool Use** | 兼容 Claude（anthropic）+ 国内/开源（OpenAI-compatible）；杜绝 prompt 注入直出 bash |
| UI | **ratatui TUI + clap CLI 双模式** | TUI 演示效果好（Modal/Tab/告警），CLI 便于 CI/脚本接入 |
| 风险判定 | **正则规则 + 工具语义双层** | 正则覆盖文本攻击，语义层补足结构化工具的真实意图 |
| 回滚机制 | **Tool 端约定 + Memory 链式记录** | 不引入分布式事务，工程量可控；apt/user/service/file 各自有反向操作模板 |
| 风险拒绝时机 | **Critical 直接拒绝 / High 强制确认** | 不给用户在 Critical 上"yes"的机会，符合安全最佳实践 |
| 审计格式 | **JSON Lines** | 易 grep/分析，评委可直接 `cat audit.jsonl \| jq` 验证 |
| 配置加载优先级 | **CLI > 配置文件 > 环境变量 > 自动探测 > 默认** | 兼顾本地开发、CI、容器三类场景 |
| 存储 | **本地 ~/.sysweaver/** | 无需数据库，单机部署即可；Playbook 用 YAML/JSON |
| 远程执行 | **SSH（OpenSSH 协议）** | 不引入自有 Agent；评委不需在远端装额外软件 |

## 七、覆盖范围与限制条件

### 7.1 覆盖能力
- 系统信息（os/cpu/mem/disk/net/process/user/service）
- 用户管理（增删改查）、服务管理、进程管理
- 包管理（apt/yum/dnf/brew 自适应）
- 文件读/写/搜索（自动备份）
- 计划任务（cron）、日志查看（log.tail）
- 网络诊断（端口/连接/DNS）
- 健康巡检（disk/memory/服务异常）
- 配置反向解释（explain 命令）
- 多步任务、撤销、Playbook

### 7.2 已知限制
- **交互式命令**：shell.exec 不支持要求 stdin 的命令（如 `passwd` 直接交互）— 通过 user.manage 提供等价能力。
- **超长输出**：工具结果超过 4000 字符时截断（避免浪费 LLM 上下文）。
- **流式响应**：当前 LLM 返回一次性接收，未做 streaming（不影响功能，仅影响首屏时延）。
- **Token 计费**：未实现 token 用量统计（演示场景下不阻塞）。
- **图形界面**：仅 TUI，不提供 Web UI。

### 7.3 安全约束（强制）
- ❌ 不允许 LLM 输出 bash 后绕过分类器执行。
- ❌ 不允许 `curl ... | bash` 形式（供应链攻击）。
- ❌ 不允许 Critical 级别走用户确认。
- ✅ Base URL 必须 HTTPS、不得嵌入凭证。
- ✅ Claude 模型只能配在 anthropic provider，保留原生 tool_use 语义。
- ✅ 所有写操作前优先 dry_run 预览。

## 九、可复现性与可审计性

1. **环境变量驱动**：`SYSWEAVER_LLM_*` 一组变量即可切换任意 Provider。
2. **审计日志**：`~/.sysweaver/audit.jsonl` 内每条记录包含 session_id、ts、tool、args、risk_level、result。
3. **会话历史**：`/history`、`/summary`、`/export` 命令一键导出可复现的对话。
4. **Playbook**：将一次成功的多步任务保存为 playbook，下次复现仅需 `sysweaver playbook run <name>`。
5. **dry-run 模式**：`--dry-run` 全程不真正执行，便于评委快速过流程。
