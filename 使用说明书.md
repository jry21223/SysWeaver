## 项目概述

**Agent Unix** 是一个用 Rust 编写的操作系统智能代理，将自然语言转换为可控的 Linux 系统操作。

**核心特性**：
- 🤖 **自然语言交互**：用中文描述任务，Agent 自动执行
- 🔧 **工具驱动架构**：所有操作通过工具调用完成，安全可控
- 🛡️ **五级风险评估**：Safe → Low → Medium → High → Critical
- 📋 **Playbook 模板**：保存常用多步骤任务，一键执行
- 📖 **反向解释模式**：读取配置文件并用自然语言解释
- 🔍 **Watchdog 监控**：后台监控资源使用，异常告警

## 核心功能详解

### Playbook（任务模板）

#### 什么是 Playbook？

**Playbook 是预定义的多步骤任务模板**，类似于 Ansible Playbook 的概念。

想象你经常需要执行一系列固定步骤的任务：
1. 检查磁盘空间
2. 检查内存使用
3. 检查关键服务状态
4. 如果发现问题，发送告警

每次都要手动输入这 4 步很麻烦。**Playbook 让你保存这套流程，下次一键执行**。

#### Playbook 的三级来源

Agent Unix 支持三种 Playbook 来源，按优先级排序：

| 来源 | 路径 | 说明 |
|------|------|------|
| **Bundled（内置）** | 系统预定义 | 3 个常用模板，无需配置 |
| **User（用户级）** | `~/.agent-unix/playbooks/*.json` | 个人常用模板 |
| **Project（项目级）** | `.agent-unix/playbooks/*.json` | 项目特定模板，**最高优先级** |

**优先级规则**：
- 同名 Playbook 时，Project 级覆盖 User 级，User 级覆盖 Bundled
- 这让你可以在项目级别定制模板，不影响全局

#### 内置 Playbook 示例

运行 `./target/release/agent-unix playbooks` 查看：

```
📋 已加载的 Playbook：

   来源统计：内置 3，用户 0，项目 0，覆盖 0

   • system-health-check — 检查系统健康状态：磁盘、内存、CPU、关键服务
     步骤数：3，运行次数：0

   • install-web-server — 安装并配置 Nginx Web 服务器
     步骤数：3，运行次数：0

   • cleanup-old-logs — 清理 30 天前的旧日志文件（安全模式）
     步骤数：3，运行次数：0
```

#### 内置 Playbook 详细说明

**1. system-health-check（系统健康检查）**

执行步骤：
```json
[
  { "tool": "system.info", "args": { "query": "disk" } },    // 检查磁盘
  { "tool": "system.info", "args": { "query": "memory" } },  // 检查内存
  { "tool": "system.info", "args": { "query": "service" } }  // 检查服务
]
```

适用场景：日常巡检、故障排查前先看整体状态。

**2. install-web-server（安装 Nginx）**

执行步骤：
```json
[
  { "tool": "shell.exec", "args": { "command": "sudo apt update && sudo apt install -y nginx" } },
  { "tool": "shell.exec", "args": { "command": "sudo systemctl start nginx && sudo systemctl enable nginx" } },
  { "tool": "system.info", "args": { "query": "service", "filter": "nginx" } }
]
```

适用场景：快速部署 Web 服务器（仅适用于 apt 系统）。

**3. cleanup-old-logs（清理旧日志）**

执行步骤：
```json
[
  { "tool": "shell.exec", "args": { "command": "find /var/log -type f -mtime +30 | head -20" } },  // 先查找
  { "tool": "shell.exec", "args": { "command": "find /var/log -type f -mtime +30 -exec ls -lh {} \\; | awk '{print $5}'" } }, // 计算大小
  { "tool": "shell.exec", "args": { "command": "find /var/log -type f -mtime +30 -delete", "dry_run": true } }  // 删除（默认预览）
]
```

注意：第三步默认 `dry_run: true`，先预览不实际删除。

#### 如何创建自定义 Playbook？

**步骤 1：在 chat 模式下完成多步骤任务**

```bash
./target/release/agent-unix chat

# 输入：
"先检查磁盘，再检查内存，最后查看 nginx 状态"

# Agent 执行后会记录操作历史
```

**步骤 2：保存为 Playbook（需要实现保存功能）**

在 chat 内输入：
```
save playbook my-health-check "自定义健康检查模板"
```

系统会将最近 3 步操作保存为 Playbook。

**步骤 3：手动创建 Playbook 文件**

创建 `~/.agent-unix/playbooks/my-playbook.json`：

```json
{
  "name": "my-custom-playbook",
  "description": "我的自定义任务模板",
  "steps": [
    {
      "tool": "system.info",
      "args": { "query": "disk" },
      "reason": "检查磁盘",
      "dry_run": false
    },
    {
      "tool": "shell.exec",
      "args": { "command": "df -h" },
      "reason": "详细磁盘信息",
      "dry_run": false
    }
  ],
  "created_at": "2026-04-21T00:00:00Z",
  "run_count": 0
}
```

**步骤 4：验证加载**

```bash
./target/release/agent-unix playbooks

# 输出应包含：
#    • my-custom-playbook — 我的自定义任务模板
#      步骤数：2，运行次数：0
```

#### Playbook 执行方式

**方式一：chat 模式内调用**

```bash
./target/release/agent-unix chat

# 输入：
"运行 system-health-check playbook"
# 或
"执行安装 Web 服务器模板"
```

**方式二：直接命令（需要实现 run-playbook 子命令）**

```bash
./target/release/agent-unix run-playbook system-health-check
```

### Agent 对话模式

#### 启动对话

```bash
./target/release/agent-unix chat
```

启动后会：
1. 采集系统环境（OS、CPU、内存、磁盘、服务）
2. 显示欢迎信息
3. 进入交互式对话循环

#### 对话内指令

| 指令 | 说明 |
|------|------|
| 自然语言任务 | "查看磁盘使用情况" |
| `history` | 查看操作历史 |
| `undo` | 撤销上一步操作（如果有回滚方案） |
| `exit` | 退出对话 |

#### 示例对话


### 反向解释模式

#### 功能说明

**反向解释模式让 Agent 读取配置文件并用自然语言解释其含义**。

适用场景：
- 新手运维不理解 nginx.conf 的配置项
- 需要快速了解某个配置文件的作用
- 文档缺失时的自助理解

#### 使用方式

```bash
# 列出支持的文件类型
./target/release/agent-unix explain

# 解释指定文件
./target/release/agent-unix explain /etc/passwd
./target/release/agent-unix explain /etc/nginx/nginx.conf
```

#### 支持的文件类型

| 类型 | 文件路径 | 说明 |
|------|----------|------|
| **配置文件** | `/etc/nginx/nginx.conf` | Nginx 配置 |
| | `/etc/ssh/sshd_config` | SSH 服务器配置 |
| | `/etc/rsyslog.conf` | 系统日志配置 |
| | `/etc/crontab` | 定时任务配置 |
| **日志文件** | `/var/log/syslog` | 系统日志 |
| | `/var/log/auth.log` | 认证日志 |
| | `/var/log/nginx/access.log` | Nginx 访问日志 |
| **系统文件** | `/etc/hostname` | 主机名 |
| | `/etc/passwd` | 用户列表 |
| | `/etc/fstab` | 挂载配置 |

#### 输出示例

```
📖 正在读取文件: /etc/passwd
   文件大小: 2048 字节

📝 文件内容（前 500 字符）：
root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
...

💡 可将以下提示词发给 Agent 进行解释：

请解释以下 用户列表文件 的内容，用简洁的中文说明：

【文件内容】
```
root:x:0:0:root:/root:/bin/bash
...
```

【解释要求】
1. 说明文件的作用和用途
2. 列出主要配置项及其含义
3. 指出潜在问题或优化建议
4. 如果是日志文件，总结最近发生的事件
```

### Watchdog 监控系统

#### 功能说明

**Watchdog 是后台监控系统，定期检查资源使用，超过阈值时发送告警**。

适用场景：
- 长时间运行的服务器需要持续监控
- 避免资源耗尽导致服务中断
- 演示"实时监控"能力

#### 默认监控规则

| 规则 | 指标 | 阈值 | 告警级别 |
|------|------|------|----------|
| 磁盘空间告警 | 磁盘使用率 | > 80% | ⚠️ Warning |
| 磁盘严重告警 | 磁盘使用率 | > 95% | 🚨 Critical |
| 内存告警 | 内存使用率 | > 85% | ⚠️ Warning |
| 内存严重告警 | 内存使用率 | > 95% | 🚨 Critical |

#### 使用方式

```bash
# 监控 60 秒（默认）
./target/release/agent-unix watch --duration 60

# 监控 10 秒演示
./target/release/agent-unix watch --duration 10
```

#### 输出示例

```
🔍 启动 Watchdog 监控系统
   监控规则：磁盘使用率（>80%/95%）、内存使用率（>85%/95%）
   监控时长：60 秒

⚠️ [WARNING] 磁盘 / 使用率已达 85.2%，超过阈值 80.0%
   当前值: 85.2%，阈值: 80.0%
   时间: 14:30:15

🚨 [CRITICAL] 内存使用率已达 96.5%，超过阈值 95.0%
   当前值: 96.5%，阈值: 95.0%
   时间: 14:30:45

✅ 监控已完成
```

## 安全机制

### 五级风险评估

Agent Unix 对所有工具调用进行风险评估：

| 级别 | 说明 | 行为 |
|------|------|------|
| **Safe** | 完全安全的只读操作 | 直接执行 |
| **Low** | 低风险，影响有限 | 直接执行 |
| **Medium** | 中风险，在 `--mode safe` 下需确认 | safe 模式确认，normal 模式直接执行 |
| **High** | 高风险，可能影响服务 | **必须用户确认** |
| **Critical** | 极危险，可能导致系统崩溃 | **直接拒绝，不给确认机会** |

### Critical 级别拦截示例

```bash
./target/release/agent-unix chat

# 输入：
"删除 /etc/passwd"

# 输出：
🚨 操作已被强制阻止（CRITICAL 级别）
原因：删除系统关键文件可能导致系统无法正常启动
影响：所有用户无法登录，系统服务异常
建议：如果需要修改用户信息，请使用 usermod 命令
```

### High 级别确认示例

```bash
# 输入：
"停止 sshd 服务"

# 输出：
⚠️   高风险操作 — 需要您确认
============================================================
工具：shell.exec
参数：{"command": "sudo systemctl stop sshd"}
风险：停止 SSH 服务可能导致无法远程登录
影响：所有 SSH 连接将被断开，无法远程管理服务器
建议：建议先确认有其他管理方式（如控制台访问）
============================================================
输入 'yes' 确认执行，其他任意键取消 › yes

✅ 成功：sshd 服务已停止
```

## 完整命令列表

### CLI 命令

| 命令 | 说明 | 需要 API Key |
|------|------|--------------|
| `chat` | 交互式对话模式 | ✅ 需要 |
| `run <指令>` | 执行单条指令 | ✅ 需要 |
| `run --dry-run <指令>` | 预览模式（不实际执行） | ✅ 需要 |
| `history` | 查看操作历史 | ❌ 不需要 |
| `undo` | 撤销上一步（需在 chat 内） | ✅ 需要 |
| `playbooks` | 列出 Playbook 模板 | ❌ 不需要 |
| `explain` | 列出支持的文件类型 | ❌ 不需要 |
| `explain <文件>` | 读取并解释配置文件 | ❌ 不需要 |
| `watch --duration <秒>` | 启动后台监控 | ❌ 不需要 |

### CLI 全局参数

```bash
agent-unix [OPTIONS] <COMMAND>

Options:
  --verbose           详细日志输出
  --mode <MODE>       运行模式：safe | normal | auto
  --provider <PROVIDER>  LLM Provider：anthropic | openai-compatible
  --model <MODEL>     LLM 模型 ID
  --base-url <URL>    LLM API Base URL
```

### 运行模式说明

| 模式 | 说明 |
|------|------|
| `safe` | 所有写操作（Medium 及以上）都需要确认 |
| `normal` | 只有 High 及以上需要确认（默认） |
| `auto` | 只有 Critical 拦截，其他自动执行 |

## 常见问题

### Q1: playbooks 命令显示"内置 3，用户 0，项目 0"，如何添加自定义 Playbook？

**答**：创建 `~/.agent-unix/playbooks/` 目录，放入 JSON 文件：

```bash
mkdir -p ~/.agent-unix/playbooks

# 创建自定义 Playbook
cat > ~/.agent-unix/playbooks/my-check.json << 'EOF'
{
  "name": "my-disk-check",
  "description": "检查磁盘并发送告警",
  "steps": [
    {
      "tool": "system.info",
      "args": { "query": "disk" },
      "reason": "检查磁盘使用率",
      "dry_run": false
    }
  ],
  "created_at": "2026-04-21T00:00:00Z",
  "run_count": 0
}
EOF

# 验证
./target/release/agent-unix playbooks
# 应显示：内置 3，用户 1，项目 0
```

### Q2: 在 macOS 上某些命令报错"systemctl: command not found"

**答**：macOS 没有 systemctl，使用 launchctl 替代。演示时建议：
- 在 Linux 服务器上运行（推荐）
- 或使用通用命令（`ps`、`top`、`df`）

### Q3: 如何在 chat 内运行 Playbook？

**答**：输入自然语言描述即可：

```
"运行 system-health-check playbook"
"执行系统健康检查模板"
"按照 system-health-check 的步骤执行"
```

Agent 会识别 Playbook 名称并执行。

### Q4: undo 撤销了什么？

**答**：Undo 根据操作类型生成回滚方案：

| 操作 | 回滚方案 |
|------|----------|
| `apt install nginx` | `apt remove nginx` |
| `useradd testuser` | `userdel testuser` |
| `systemctl start sshd` | `systemctl stop sshd` |
| `rm file.txt` | ❌ 无法自动回滚（需要备份） |
| `file.write` | ❌ 无法自动回滚（需要备份原内容） |

### Q5: 如何配置自定义 LLM 端点？

**答**：使用环境变量或 CLI 参数：

```bash
# OpenRouter 示例
export AGENT_UNIX_LLM_PROVIDER=openai-compatible
export AGENT_UNIX_LLM_BASE_URL=https://openrouter.ai/api
export AGENT_UNIX_LLM_MODEL=openai/gpt-4o
export AGENT_UNIX_LLM_API_KEY=sk-or-你的key

# 或 CLI 参数
./target/release/agent-unix \
  --provider openai-compatible \
  --base-url https://openrouter.ai/api \
  --model openai/gpt-4o \
  chat
```
